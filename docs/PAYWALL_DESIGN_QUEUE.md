## Q2. 수신 지갑은 누구의 것인가

결제 USDC가 어디로 가야 하는가:

- **(a) 파일 소유자 지갑으로 직접** — P2P, 가장 단순. 소유자가 미리 payout wallet을 등록해야 함 (현재 `app/api/wallet/me`로 가능해 보임).
- **(b) 플랫폼 지갑** → 정산 — 수수료 차감, 환불 처리 가능, 다만 escrow/회계 부담.
- **(c) 하이브리드** — 기본은 (a), 플랫폼 수수료는 별도로 처리.

비고: 현재 `app/api/s/[token]/pay/route.ts`는 `process.env.AINDRIVE_PAYOUT_WALLET` 단일 환경변수로 처리 중 — 즉 모든 드라이브가 한 곳으로 송금되는 (b)에 가까운 임시 구현.

## Q3. 결제자도 지갑을 연결해야 하나

영구 access를 어떤 식별자에 묶을지:

- **지갑 주소 기준** — 가장 자연스럽고 web3-네이티브. 결제자도 지갑 연결 필수.
- **aindrive 계정 기준** — 지갑 결제 + 계정 식별 둘 다 필요해서 복잡.
- **혼합** — 영구 결제는 지갑 연결 강제, 1회 결제는 익명 지갑 서명만 OK.

비고: 현재 `app/api/s/[token]/pay/route.ts`가 이미 지갑 쿠키를 세션에 바인딩하는 패턴 (`setWalletCookie`). 그리고 `folder_access` 테이블이 `wallet_address`를 키로 씀 → 지갑 기준 모델이 이미 코드에 박혀있음.

## Q4. 파일 삭제·가격 변경·환불 정책

- **소유자가 파일을 삭제하면**: 영구 결제로 산 사람의 권한은 어떻게 되는가?
  - 권한은 유지하되 404를 받게 됨 (단순)
  - 자동 환불 (체인상 환불은 facilitator 지원 미흡 — 사실상 어려움)
  - 정책으로 "삭제 금지" 또는 "삭제 시 경고" UI
- **소유자가 가격을 올리면**: 기존 구매자에게 영향 없음 (자명)
- **소유자가 가격을 0으로 내리면 (무료화)**: 환불해야 하는가? 일반적으로 No.
- **환불·분쟁 정책**: v1은 "모든 결제는 최종, 환불 없음"으로 시작 권장.

## Q5. 잠금 표시 정밀도

비-소유자가 디렉터리 리스팅할 때 유료 파일을 어떻게 보여줄 것인가:

- **(a) 파일명/크기/유형 모두 보임, 내용만 잠김** — 미리보기로 "이 파일이 뭘 살건지" 짐작 가능.
- **(b) 파일명만 가림** — 썸네일/내용 없음.
- **(c) 존재만 표시** — "잠긴 파일 N개".

추가: 가격을 리스팅에 함께 노출할 것인가? 호버/클릭 시 결제 모달에서 노출할 것인가?

## 1회 결제의 의미 (별도 트랙)

x402와의 적합성이 가장 큰 미해결 항목:

- **(a) 그 한 번의 다운로드만** — x402 가장 자연스러움. 재시도 방법 없음.
- **(b) 결제 후 짧은 윈도우(60초~10분) 동안 접근 허용** — UX 안전. DB에 만료시각 기록.
- **(c) "한 번 본다 = 화면에 띄운다"** — 사실상 시행 불가능 (스크린캡처/저장).
