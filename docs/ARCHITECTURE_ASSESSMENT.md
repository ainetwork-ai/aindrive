# aindrive: Hackathon → Product Architecture Roadmap

## 1. Verdict

Architecturally, aindrive is a **strong hackathon spike with a clean conceptual core, roughly one focused quarter away from being a real product** — but only if the foundation is fixed *before* features pile on. The load-bearing decisions (single-source permission model, append-only payment ledger, agent-as-peer RPC, isolated packages) are sound and product-grade. What stands between it and "real product" is not the vision but three structural debts: a **single-process-in-memory lock-in** baked into every realtime/state path, a **schema-and-protocol "single source of truth" that has silently forked into 3-5 copies**, and the fact that **the headline product promise — selling copyright, not just access — has zero data model behind it.** None of these are rewrites; all are foundational and must be sequenced deliberately.

## 2. Already solid — do not churn

These are product-grade. Resist the urge to refactor them while doing P0 work.

- **The permission model (`drive_members` single source).** `access-core.js` (pure, DB-free, shared by raw-Node `dochub.js` and Next `access.ts`), `require-access.ts` gate factory, and the settle/consume paths all funnel through one source. This is the redesign you said to treat as solid — it genuinely is. Build *on* it.
- **The `.js` + `.d.ts` dual-runtime pattern** for `access-core` and `path` — the one place a "duplicate" is justified (one file must run under both raw `server.js` and the Next TS pipeline). Correct and battle-tested. Distinguish this from the *unjustified* duplication (sig, protocol, schema).
- **Payment integrity primitives.** Append-only `payment_receipts`, `tx_hash` UNIQUE replay guard, `account_wallets` placeholder-account bridge (buyers can pay before they have a login and reclaim later), `mergeRoleUpgradeOnly`, crash-safe post-settle write. The *ledger* is right; what's missing is layered on top, not underneath.
- **Boot-checks** (verified present): the four P0 landmines (bypass-in-prod, weak/missing session secret, non-HTTPS URL, zero/missing payout-wallet env) all `process.exit(1)` in production. The payments audit's "DEV_BYPASS has no boot guard" smell is **stale** — that guard exists. (The remaining gap is the *per-drive* zero-wallet fallback at settle time, which env-level boot-checks don't cover.)
- **Lean `server.js`** (58 lines, delegates + graceful shutdown) and **graceful-drain shutdown** on both server and CLI.
- **`web/src/composition.ts` DI root** — the ports-and-adapters pattern, where it exists, is done correctly. The problem is it's half-applied, not that it's wrong.
- **The package boundary discipline** (web/cli isolation, no root shared code) is a deliberate, documented choice — keep it.

## 3. Cross-cutting themes

These are the issues that recur across dimensions. Each is foundational because it gates multiple features at once.

### Theme A — Single-process / in-memory lock-in is everywhere
**Touches:** Realtime, Deploy/Observability, Identity/auth, Package structure.
Every piece of live state is a `globalThis`-pinned Map: WS agent registry (`agents.js`), DocHub subscribers (`dochub.js`), SIWE nonces and rate-limit buckets (`wallet.ts`, `rate-limit.js`). This is the *same root cause* showing up as five different "smells." **Why foundational:** it silently sets a hard ceiling at one process. The moment you add a second container, blue/green, or rolling deploy, agents become invisible across instances, DocHub stops fan-out, rate limits multiply by N, and nonces become replayable on the other node. This single decision is the largest fork in the whole roadmap (see Open Decision 1) — almost everything in scaling depends on it.

### Theme B — "Single source of truth" has silently forked
**Touches:** Package structure, Data model, CLI, Testing.
The codebase *claims* canonical sources but operates on copies: **5 copies of the sig function**, **2+ diverged `protocol.ts` files** (the shared one a stale draft missing the `yjs-*`/`agent-ask` methods the system actually uses), **3-way schema drift** (`schema.ts` / `schema.js` / raw DDL in `db.js`), and tables (`cli_link_requests`) that exist only in raw SQL. **Why foundational:** these are exactly the files where silent drift is most dangerous — security crypto, wire protocol, and DB schema. Every one will eventually be edited in one place and missed in another, and the failure mode is silent (wrong column returns `undefined`, a replayed sig still verifies). This is also a *velocity tax*: it's the single biggest thing making the repo hostile to a growing team.

### Theme C — The copyright-sale layer is entirely absent
**Touches:** Payments (primarily), Data model, Frontend, CLI/MCP.
The stated #1 product goal is selling **content AND copyright**. Today a purchase writes a `drive_members` viewer row — *byte-identical to a free collaborator invite.* There is no `license_type`, no rights snapshot on the receipt, no buyer-accessible proof-of-purchase, no resale/chain-of-title. **Why foundational:** this isn't a feature you bolt on later — it's the product's reason to exist, and it determines the shape of the receipts schema, the share-creation API, the sell UI, and eventually the on-chain rights representation. Shipping payments without it means shipping a generic paywall and calling it a copyright marketplace.

### Theme D — Migration story is aspirational, not operational
**Touches:** Data model, Deploy, Testing.
Schema today is `CREATE TABLE IF NOT EXISTS` + an idempotent `ALTER` loop that string-matches error text, plus two hand-written one-shot scripts. `drizzle-kit` is *configured but not installed* — false scaffolding. There's no version tracking, no rollback, no CI gate against a realistic pre-migration DB. **Why foundational:** every future schema change (and Themes C and B both *require* schema changes) rides on this. Until there's a real migration framework with a version table, every column add post-launch is a high-risk manual operation, and the documented Postgres path (Open Decision 2) is unreachable.

### Theme E — Test & observability vacuum on exactly the money/realtime paths
**Touches:** Testing, Deploy/Observability, Payments, Realtime, Security.
The ~160-case scenario suite (which covers share/payment, Yjs, Willow sync) is **permanently excluded from CI** and hardcoded to one developer's `/mnt/newdata/...` paths. The CLI has **zero tests**. There's no Sentry, no log sink, no metrics, no off-host backup. **Why foundational:** you cannot safely execute Themes A–D without a net. Refactoring in-memory state to a shared store, migrating the schema, or adding a license layer to the settle path are all changes that *will* break something — and right now CI stays green while the paid-share flow breaks, and production gives you no signal when it does.

### Theme F — Security gaps cluster on the unauthenticated edges
**Touches:** Security, Deploy, Realtime.
A tight cluster of real holes, all on internet-facing surfaces: SIWE nonce via `Math.random()`, `wallet/verify` not binding domain (phishing-replay), **unauthenticated `/api/dev/trace/dump`** leaking the full operation log + userIds in prod, CSP neutered by `unsafe-inline`/`unsafe-eval`, unsigned `sync-*` WS relay, no rate limit on the money endpoint `/api/s/[token]`. **Why foundational:** these are cheap to fix and catastrophic to leave, and several (nonce, trace endpoint, money-endpoint rate limit) become *more* exposed the moment you go to mainnet with real funds.

## 4. Prioritized roadmap

Sequencing is deliberate: P0 builds the net + the canonical sources, P1 makes the product honest and safe for real money, P2 unlocks scale and the realtime vision, P3 is the long game. **Do not start P1's license layer before P0's migration framework exists — they share the schema-change machinery.**

---

### P0 — Foundation: net, canonical sources, edge security *(do first; everything rides on this)*
*Why now: you said "fix the architecture first." This is the smallest set of moves that makes every later change safe and de-risks the rest. Mostly mechanical, low-conceptual-risk.*

1. **Collapse the forked sources of truth** (Theme B).
   - Sig → one file (`web/shared/crypto/sig.ts`); others re-export or hand-mirror with a `// Mirrors` comment; delete the rest.
   - Make `web/shared/protocol.ts` the real operative protocol (add `yjs-write/read`, `agent-ask`, `yjs-stats`); delete `web/lib/protocol.ts`; CLI imports/mirrors from the canonical with one comment.
   - Kill `schema.js`: emit it from `schema.ts` at build, or import the `.ts` through the pipeline. Add `cli_link_requests` to the schema.
   *Effort: S–M. Pure consolidation, no behavior change.*

2. **Real migration framework** (Theme D). Install `drizzle-kit`, add `db:generate`/`db:migrate` scripts and a `__drizzle_migrations` table, generate a baseline from current `schema.ts`, convert 0001/0003. Replace the startup `ALTER` loop. Add a `role` CHECK constraint while you're in there.
   *Effort: M. **This is the prerequisite for P1's license columns.***

3. **CI net on the paths that matter** (Theme E). De-hardcode the scenario suite (`AINDRIVE_SAMPLE_DIR`, `AINDRIVE_CLI_SRC` env), add a containerized CI job (web server + agent on tmp dirs) that runs scenarios as a merge gate. Add a minimal CLI test suite (`safeResolve` traversal, HMAC round-trip, yjs round-trip). Pin CI to Node 22 to match Docker, or add a `docker build` gate.
   *Effort: M–L.*

4. **Close the edge-security cluster** (Theme F) — all small, all high-severity: `crypto.randomBytes` nonce; domain-bind `wallet/verify`; auth-gate `/api/dev/trace/*`; rate-limit `/api/s/[token]`, `/api/wallet/nonce`, `/api/auth/cli/poll`; reduce `serverActions.bodySizeLimit` to the write cap; HMAC-verify the `sync-*` relay.
   *Effort: S. A day or two; do not skip — these get worse at mainnet.*

5. **Off-host backup + error tracking** (Theme E) — litestream sidecar streaming SQLite to S3-compatible storage; Sentry on web + CLI; `HEALTHCHECK` in Dockerfile; fix `/readyz` to gate on `app.prepare()`.
   *Effort: S–M. No code architecture change; this is the "don't lose all user funds/data" floor.*

---

### P1 — Make the product honest: the copyright-sale layer + mainnet *(needs P0 #2 migrations)*
*Why now: this is the actual product. But it literally cannot be done safely until the migration framework (P0 #2) exists, because it's all schema changes on the money path — which is also why it's P1 not P0.*

1. **License/rights data model** (Theme C). Add `license_type` to `shares` and snapshot it onto `payment_receipts` at settle time. This is the minimal change that makes "copyright sale" a product rather than marketing copy. Sell UI gets a dropdown.
2. **Network as runtime config** — single `AINDRIVE_PAYMENT_NETWORK` env (`base` | `base-sepolia`) driving the route constant, wagmi chain, USDC address, and Basescan URL; add to boot-checks. Removes the multi-file code-change-to-go-mainnet coupling.
3. **Marketplace minimum viable controls** — share-disable (`active` flag + kill switch), per-drive payout-wallet *enforcement* (reject settle to zero address, not silent burn), buyer-facing `GET /api/receipts` (proof-of-purchase), `price_usdc` → integer micro-USDC.
4. **Fix the Meadowcap cap delivery bug** — the buyer currently receives `capBase64` but not `recipientSecret`, making the cap useless. Deliver both, or descope the cap layer until WGPS is real (see Open Decision 4).
5. **Audit log + signup email verification** (Theme F, compliance floor for real money).
   *Effort: L overall, but each item is M or smaller. Sequence #1 and #2 first — they unblock a real launch.*

---

### P2 — Unlock scale & the realtime vision *(the big fork lives here)*
*Why now: only after the product is real and tested. This is where Theme A (single-process lock-in) gets resolved — and it's the most expensive, most reversible-if-wrong decision, so it deserves the runway P0/P1 buy you.*

1. **Resolve the in-memory-state lock-in** per Open Decision 1. If scaling out: Redis pub/sub for DocHub fan-out + agent routing; shared-store nonces/rate-limits; server-side Y.Doc persistence + catch-up for late joiners.
2. **Permission revocation on live WS subscribers** (today revocation only takes effect on next subscribe — unacceptable once a sold share can be revoked).
3. **Unify cap-bearer + session auth in the WS upgrade path** so paid Meadowcap collaborators can use the editor.
4. **Finish or kill `web/src/`** (Theme B/structure) — commit to ports-and-adapters and migrate the rest, or fold the clean use-cases back into `lib/` and delete the subtree. Half-migration is the worst state; this is a decision, not just work.
   *Effort: L. Decision 1 dominates.*

---

### P3 — Long game
**WGPS / real disk-backed Willow store is the committed end-state architecture** (Decision 4), not optional: server-side disk payload driver (replace the hollow `PayloadDriverMemory`) + WGPS P2P capability sync + binary WS frames, delivering agent-optional browser access. P0's Willow cleanup is explicitly path-preserving so this is an *extension*, not a rewrite (keep the entry-driver/schema/Meadowcap; only the payload driver, the bypassed write-auth, and the server-side sync endpoint remain to build). Also here: on-chain license representation (ERC-1155 / resale / chain-of-title — the on-chain half of Decision 3), Postgres dialect switch if Decision 2 ever flips, metrics/Prometheus + load testing, frontend polish (error boundaries, replace native `prompt/confirm`, a11y, TanStack Query for reads). *Effort: XL, ongoing.*

## 5. First move

**Stand up the CI net on the money + realtime paths (P0 #3), and do P0 #1 (collapse sig/protocol/schema) in the same first push.**

Why this and not the security fixes or the license layer: every other foundational move — migrations, the license schema, the in-memory→shared-store refactor — is a change that *will* break something on a path that currently has **no automated coverage and no production error signal.** Right now you can break the paid-share flow or Yjs convergence and CI stays green. You cannot responsibly "fix the architecture first" without first being able to *see* when a fix breaks something. Pairing it with the source-of-truth consolidation means your very first refactor (deleting 4 sig copies, unforking protocol) is itself validated by the net you just built — a clean proof the net works, on a low-risk change. Security edge-fixes (P0 #4) are a close second and should land in the same P0 window, but they don't *unblock* the rest the way the net does.

## 6. Decisions — RESOLVED (2026-06-05)

All five forks are settled. Summary of the chosen directions and their end-states:

| # | Decision | Now | End-state |
|---|---|---|---|
| 1 | Instance model | Single-process, but live state behind `PresenceStore`/`NonceStore`/`RateLimiter` interfaces (Map impls) + multi-process-refusal startup guard | Swap interface impls to Redis → horizontal scale (no rewrite) |
| 2 | Database | SQLite + litestream off-host backup; Drizzle schema authored dialect-portable | Flip Drizzle dialect → Postgres (config, not rewrite) |
| 3 | Copyright/license layer | Deferred — purchase = paid access only (no `license_type` yet) | On-chain rights (ERC-1155 / resale / chain-of-title) at P3 |
| 4 | Willow/WGPS | P0 path-preserving cleanup (kill hollow dual-write, fix/quarantine cap delivery; KEEP entry-driver/schema/Meadowcap) | **Committed:** server-side disk store + WGPS = agent-optional browser access (P3) |
| 5 | `web/src/` architecture | Finish ports-and-adapters (after P0 net; only at real I/O seams) | Ports-and-adapters is the convention; `web/lib/` → shared infra only |

Common thread: **abstract now, swap the implementation later** — every deferral leaves an explicit, recorded path to its end-state so nothing is silently lost. Details below.

1. **Single-instance vs. design for horizontal scale.** *Stay single-process* (one container, vertical scale, `globalThis` maps are fine, ship faster, accept a hard ceiling and downtime on deploy) **vs.** *design for scale-out now* (Redis-backed state, server-side Y.Doc, ~weeks of work, but blue/green + HA become possible). Tradeoff: nearly every Theme-A item is "fine" under the first and "blocking" under the second. **This is the master fork — it sets the cost of half the roadmap.** Honest take: a writer/artist marketplace at launch volume can live single-instance for a while; just make the choice *explicitly* and add a startup guard that refuses multi-process, rather than drifting into it.

2. **SQLite vs. Postgres.** *Stay on SQLite* (litestream backup, single-writer ceiling, zero infra, perfect for single-instance) **vs.** *move to Postgres* (real concurrency, PITR/WAL archiving, multi-instance-ready, but ops overhead + the migration framework must target it from day one). Tradeoff: tightly coupled to Decision 1. If you stay single-instance, SQLite is genuinely fine and over-migrating is waste. If you go scale-out, the Drizzle schema should be authored dialect-portable *now* so the switch is config, not rewrite.

3. **How literal is the copyright/licensing layer at launch?** *Off-chain license records* (a `license_type` column + a buyer-fetchable receipt = "you bought commercial-use rights," operator-attestable) **vs.** *on-chain rights* (ERC-1155/NFT representing the license, enabling true resale and verifiable chain-of-title, but a much larger build and a regulatory surface). Tradeoff: off-chain is honest enough to launch and is the P1 move above; on-chain is the differentiator but is P3. The fork is *how much you promise buyers at launch* — be careful not to market "own the copyright, resell it" before the on-chain layer exists.

4. **Willow/WGPS — RESOLVED: WGPS is the committed END-STATE goal; sequenced to P3, NOT abandoned.** The destination is a server-side disk-backed Willow store + WGPS P2P capability sync, so a browser (or a buyer) can access drive contents **without the owner's CLI agent being online** — the full "your local folder, on the web, even when your laptop's closed" promise. That is the target architecture (also in `docs/WILLOW_DESIGN.md`); it is a commitment, not a maybe.
   - **Interim (P0) = path-PRESERVING cleanup, not descope-as-abandon.** Today Willow is half-wired and confusing: the Store's payloads are RAM-only (`PayloadDriverMemory`) so real data rides a SQLite mirror (`yjs_entries`) while the Willow Store is dead-weight dual-write; the write-auth scheme is bypassed; the paid-share cap is undeliverable (buyer gets `capBase64` but not `recipientSecret`). P0 cleans this **while keeping the road to WGPS open**: KEEP the reusable foundation (`EntryDriverKvStore`, the Willow schema/namespace keys, Meadowcap issuance), STOP the hollow dead-weight dual-write, and either fix the cap-delivery bug or quarantine the cap path behind a clearly-marked "pre-WGPS" boundary. Do **not** rip out the entry-driver/schema that WGPS will build on.
   - **Only open sub-question is TIMING, not direction:** is "agent-optional browser access" a *launch* claim (→ pull WGPS toward P2) or *post-launch* (→ P3)? Default = P3: accept the "agent-must-be-online" limitation at launch and revisit after P0, once real usage shows how often creators stay online. Until WGPS ships, do not market "access even when the creator is offline."

5. **`web/src/` ports-and-adapters — RESOLVED: FINISH the migration (commit to ports-and-adapters as the convention).** Move auth/shares/wallet/fs into the `web/src/` use-cases + adapters structure, with `composition.ts` as the single DI root, and delete the back-references from `web/src/` into `web/lib/`. `web/lib/` shrinks to genuinely-shared infra (db, env, pure helpers) consumed via ports. Two conditions on HOW:
   - **Sequence AFTER the P0 test net.** Restructuring this much code without coverage will silently break something. The `web/src/` finish is a P1/P2 move, not P0 — P0 (test net + canonical-source consolidation) lands first so this migration is done under a net.
   - **Apply only at real I/O seams; no ceremony.** Ports/adapters pay off where there's an external dependency worth mocking — CLI RPC, DB, payments/x402, Meadowcap, sessions. Wrap those; do NOT layer abstractions around pure functions. The payoff is testability (directly attacks the Theme-E test vacuum: use-cases become unit-testable with mock adapters), not architectural purity for its own sake.
   - Unlike Decision 4, both forks here are reachable end-states, so "finish now" IS the destination — no later re-split.
